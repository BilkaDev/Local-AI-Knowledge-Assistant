What problem does retrieval solve in a RAG system?	Jaki problem rozwiązuje retrieval w systemie RAG?	Retrieval finds the most relevant document chunks before generation, so the model answers from evidence instead of memory. This reduces hallucinations and improves source traceability.	Retrieval znajduje najbardziej trafne fragmenty dokumentów przed generacją, więc model odpowiada na podstawie dowodów, a nie pamięci. To zmniejsza halucynacje i poprawia śledzenie źródeł.
When is retrieval a bad architectural choice?	Kiedy retrieval jest złym wyborem architektonicznym?	It can be overkill for small creative bots that do not need citations or factual grounding. It also adds latency and operational complexity that may not pay off.	Może być przerostem formy dla małych kreatywnych botów, które nie potrzebują cytowań ani ugruntowania faktów. Dodaje też opóźnienia i złożoność operacyjną, które mogą się nie zwrócić.
Why return source metadata like source_file and chunk_id, not only text?	Dlaczego zwracać metadane źródła jak source_file i chunk_id, a nie tylko tekst?	Metadata makes answers auditable and easy to verify in UI or logs. It also enables debugging when a chunk is wrong or outdated.	Metadane czynią odpowiedzi audytowalnymi i łatwymi do weryfikacji w UI lub logach. Umożliwiają też debugowanie, gdy chunk jest błędny lub nieaktualny.
How would you choose an initial top_k value?	Jak dobrać początkową wartość top_k?	Start with a moderate value (for example 3-5), then tune using retrieval quality and latency metrics. Too low misses context; too high injects noise.	Zacznij od umiarkowanej wartości (np. 3-5), a potem stroń na podstawie jakości retrieval i metryk opóźnień. Zbyt niskie gubi kontekst, zbyt wysokie dodaje szum.
What is the main trade-off when increasing top_k?	Jaki jest główny trade-off przy zwiększaniu top_k?	Higher top_k usually improves recall but can reduce precision by adding irrelevant chunks. It also increases token usage and response time.	Wyższe top_k zwykle poprawia recall, ale może obniżyć precyzję przez dodanie nieistotnych chunków. Zwiększa też zużycie tokenów i czas odpowiedzi.
If answers become slower, what should telemetry tell you first?	Jeśli odpowiedzi zwalniają, co telemetry powinno pokazać najpierw?	It should separate retrieval latency from generation latency, so you know where the bottleneck is. Without this split, optimization is guesswork.	Powinno rozdzielać opóźnienie retrieval od opóźnienia generacji, żeby wskazać wąskie gardło. Bez tego optymalizacja to zgadywanie.
Why is logging returned_chunks important besides timing?	Dlaczego logowanie returned_chunks jest ważne poza samym czasem?	It shows whether the retriever actually found enough context for the requested top_k. A fast query with zero useful chunks is still a failure.	Pokazuje, czy retriever faktycznie znalazł wystarczający kontekst dla żądanego top_k. Szybkie zapytanie z zerem użytecznych chunków to nadal porażka.
What error signal suggests retrieval/index mismatch?	Jaki sygnał błędu sugeruje rozjazd retrieval i indeksu?	Empty or low-quality results for clearly answerable questions often indicate indexing or metadata mismatch. Missing expected fields or wrong collection settings are common causes.	Puste lub słabe wyniki dla pytań, na które powinny być odpowiedzi, często wskazują na rozjazd indeksowania lub metadanych. Częste przyczyny to brak pól kontraktu lub zła konfiguracja kolekcji.
How should the system behave when the collection is empty?	Jak system powinien się zachować, gdy kolekcja jest pusta?	It should return a safe empty result with clear diagnostics, not crash. This keeps downstream generation predictable and easier to handle.	Powinien zwrócić bezpieczny pusty wynik z czytelną diagnostyką, a nie się wywrócić. Dzięki temu dalsza generacja jest przewidywalna i łatwiejsza do obsługi.
Why add a CLI retrieval smoke test before UI integration?	Dlaczego dodać CLI smoke test retrieval przed integracją z UI?	CLI isolates retrieval from UI complexity and speeds up debugging. It gives a repeatable way to validate behavior in local and Docker environments.	CLI izoluje retrieval od złożoności UI i przyspiesza debugowanie. Daje powtarzalny sposób walidacji działania lokalnie i w Dockerze.
What does a stable retrieval contract enable for Day 6 generation?	Co stabilny kontrakt retrieval umożliwia w Dniu 6 (generacja)?	It provides predictable input shape for prompt building, citations, and fallback logic. This reduces integration risk and test flakiness.	Zapewnia przewidywalny format wejścia dla budowania promptu, cytowań i logiki fallback. To zmniejsza ryzyko integracji i niestabilność testów.
Why test invalid top_k and empty results, not only happy paths?	Dlaczego testować nieprawidłowe top_k i puste wyniki, a nie tylko happy path?	Because failure modes happen in production and define system reliability. Guardrail tests prevent silent regressions after refactors.	Ponieważ tryby awarii zdarzają się na produkcji i definiują niezawodność systemu. Testy zabezpieczające zapobiegają cichym regresjom po refaktorach.
How does retrieval support portfolio value, not only functionality?	Jak retrieval wspiera wartość portfolio, a nie tylko funkcjonalność?	It demonstrates engineering maturity: traceability, measurable latency, and test coverage. Recruiters can see decisions, trade-offs, and operational thinking.	Pokazuje dojrzałość inżynierską: śledzenie źródeł, mierzalne opóźnienia i pokrycie testami. Rekruter widzi decyzje, trade-offy i myślenie operacyjne.
If you must optimize first, what should you tune before rewriting architecture?	Jeśli musisz najpierw optymalizować, co stroić przed przepisywaniem architektury?	Tune top_k and retrieval settings using telemetry, then reassess quality and latency. Premature redesign without metrics often wastes effort.	Stroń top_k i ustawienia retrieval na podstawie telemetry, a potem ponownie oceń jakość i opóźnienia. Przedwczesny redesign bez metryk często marnuje wysiłek.

Why is ingestion alone insufficient for practical RAG?	Dlaczego samo ingestion nie wystarcza do praktycznego RAG?	Ingestion prepares chunks but does not rank them by semantic relevance to a question. Without retrieval over embeddings, the model gets weak context and hallucinates more.	Ingestion przygotowuje chunki, ale nie ustala ich trafności semantycznej względem pytania. Bez retrieval na embeddingach model dostaje słaby kontekst i częściej halucynuje.
What core value do embeddings add in a document QA system?	Jaką kluczową wartość dają embeddingi w systemie QA na dokumentach?	Embeddings let you match by meaning, not exact wording. This increases recall when user phrasing differs from document phrasing.	Embeddingi pozwalają dopasowywać po znaczeniu, a nie po identycznych słowach. To zwiększa trafność, gdy użytkownik i dokument używają innego sformułowania.
How would you explain ChromaDB to a beginner in one practical sentence?	Jak jednym praktycznym zdaniem wyjaśnić ChromaDB początkującemu?	ChromaDB stores vectorized chunks and quickly returns the most similar ones for a query. It is the retrieval engine that feeds grounded context to the LLM.	ChromaDB przechowuje zwektoryzowane chunki i szybko zwraca najbardziej podobne do zapytania. To silnik retrieval, który dostarcza LLM-owi ugruntowany kontekst.
When is top_k too low, and what symptom appears?	Kiedy top_k jest za niskie i jaki pojawia się objaw?	top_k is too low when relevant evidence is missing from context. A common symptom is incomplete answers despite correct data being in the corpus.	top_k jest za niskie, gdy brakuje kluczowych fragmentów w kontekście. Typowy objaw to niepełne odpowiedzi mimo że poprawne dane są w bazie.
When is top_k too high, and what symptom appears?	Kiedy top_k jest za wysokie i jaki pojawia się objaw?	top_k is too high when context becomes noisy and dilutes relevance. Symptoms include slower responses and less focused answers.	top_k jest za wysokie, gdy kontekst robi się zaszumiony i traci trafność. Objawy to wolniejsze odpowiedzi i mniej precyzyjne odpowiedzi.
Why keep metadata together with embeddings?	Dlaczego trzymać metadane razem z embeddingami?	Metadata gives source traceability, citation, and debugging power. Without it, you may retrieve text but cannot reliably explain where it came from.	Metadane dają śledzenie źródła, cytowanie i łatwiejszy debugging. Bez nich możesz znaleźć tekst, ale nie wyjaśnisz wiarygodnie skąd pochodzi.
What risk does deterministic ID mitigate during re-indexing?	Jakie ryzyko redukuje deterministyczne ID przy reindeksacji?	It prevents duplicate growth across repeated runs by targeting the same record key. This keeps collection size and behavior stable over time.	Zapobiega narastaniu duplikatów przy kolejnych uruchomieniach, bo trafia w ten sam klucz rekordu. Dzięki temu rozmiar i zachowanie kolekcji pozostają stabilne.
When would deterministic IDs be a poor choice?	Kiedy deterministyczne ID byłoby słabym wyborem?	They are a poor fit when you intentionally want full historical snapshots as separate records. In that case, versioned or run-specific IDs may be better.	Są słabym wyborem, gdy celowo chcesz trzymać pełną historię jako osobne rekordy. Wtedy lepsze mogą być ID wersjonowane lub zależne od runa.
Why use contract-first validation before embedding?	Dlaczego stosować walidację contract-first przed embeddingiem?	It catches malformed records early and protects downstream quality. You trade a bit more implementation effort for much clearer failure handling.	Wcześnie wykrywa uszkodzone rekordy i chroni jakość dalszego pipeline’u. Płacisz trochę większym nakładem implementacji, ale zyskujesz dużo czytelniejsze błędy.
What is the main trade-off of skip-and-continue failure policy?	Jaki jest główny trade-off polityki skip-and-continue?	You keep pipeline continuity but risk silent quality drift if failures are not monitored. It requires discipline in reading failure reports.	Zachowujesz ciągłość działania pipeline’u, ale ryzykujesz cichy spadek jakości, jeśli nikt nie monitoruje failures. To wymaga regularnej analizy raportów błędów.
Why is “completed” not enough as a success signal?	Dlaczego „completed” nie jest wystarczającym sygnałem sukcesu?	A process can complete with zero vectors saved, meaning retrieval is still unusable. You need outcome metrics, not only process status.	Proces może się zakończyć przy zerowej liczbie zapisanych wektorów, więc retrieval dalej nie działa. Potrzebujesz metryk wyniku, a nie tylko statusu wykonania.
Which post-run gate best protects Day 4 readiness?	Która bramka post-run najlepiej chroni gotowość Dnia 4?	Checking collection_count > 0 is a strong minimal gate. Combined with vectors_saved and failures, it quickly confirms retrieval readiness.	Sprawdzenie collection_count > 0 to mocna minimalna bramka. Razem z vectors_saved i failures szybko potwierdza gotowość do retrieval.
Why did localhost fail in Docker app-to-ollama communication?	Dlaczego localhost nie zadziałał przy komunikacji app->ollama w Dockerze?	In a container, localhost points to that same container, not another service. You must use service DNS like http://ollama:11434.	W kontenerze localhost wskazuje na ten sam kontener, a nie inną usługę. Trzeba użyć DNS usługi, np. http://ollama:11434.
What is a practical connectivity error signal for embedding backend issues?	Jaki jest praktyczny sygnał błędu łączności z backendem embeddingów?	“Failed to connect to Ollama” plus vectors_saved=0 is a strong signal. It usually indicates wrong base URL, service down, or network mismatch.	„Failed to connect to Ollama” razem z vectors_saved=0 to mocny sygnał. Zwykle oznacza zły base URL, niedziałającą usługę albo błąd sieci.
How does a feedback loop improve an AI integration workflow?	Jak feedback loop poprawia workflow integracji AI?	Each run should produce measurable outputs, then adjustments follow from those signals. This reduces guesswork and speeds up quality improvement.	Każdy run powinien dawać mierzalne wyniki, a zmiany powinny wynikać z tych sygnałów. To ogranicza zgadywanie i przyspiesza poprawę jakości.
What is plan mode useful for in AI-assisted development?	Do czego przydaje się plan mode w pracy z AI?	Plan mode clarifies scope, decisions, and acceptance criteria before coding. It lowers rework risk when tasks have multiple valid implementation paths.	Plan mode porządkuje zakres, decyzje i kryteria akceptacji przed kodowaniem. Zmniejsza ryzyko poprawek, gdy zadanie ma wiele sensownych ścieżek realizacji.
When is plan mode a bad choice?	Kiedy plan mode jest złym wyborem?	It is inefficient for tiny, obvious tasks where implementation is immediate. Over-planning can delay delivery without improving quality.	Jest nieefektywny przy małych i oczywistych zadaniach, które można od razu wykonać. Nadmierne planowanie opóźnia dowóz bez poprawy jakości.
What is context engineering in agent prompting?	Czym jest context engineering w promptowaniu agenta?	It is deliberate selection of relevant constraints, files, goals, and success signals for the model. Better context usually beats longer prompts.	To świadomy dobór istotnych ograniczeń, plików, celów i sygnałów sukcesu dla modelu. Lepszy kontekst zwykle działa lepiej niż dłuższy prompt.
What is the biggest context-engineering mistake?	Jaki jest największy błąd context engineering?	Providing too much irrelevant context that buries critical constraints. This increases confusion and lowers output reliability.	Dostarczanie zbyt dużej ilości nieistotnego kontekstu, który przykrywa krytyczne ograniczenia. To zwiększa chaos i obniża wiarygodność wyników.
How should AI instructions be structured for reliable execution?	Jak strukturyzować instrukcje dla AI, żeby były niezawodne?	State goal, constraints, input artifacts, output format, and done criteria explicitly. Ambiguity should be replaced by measurable checks.	Jasno podaj cel, ograniczenia, artefakty wejściowe, format wyjścia i kryteria ukończenia. Niejednoznaczność zastępuj mierzalnymi checkami.
How do pattern-based prompts like “add feature X based on Y” help?	Jak pomagają prompty wzorcowe typu „dodaj funkcję X na podstawie Y”?	They anchor generation to existing structure and reduce inconsistent design choices. This is useful when extending systems with established conventions.	Kotwiczą generowanie w istniejącej strukturze i ograniczają niespójne decyzje projektowe. To szczególnie pomaga przy rozwijaniu systemów z ustalonymi konwencjami.
When can “add X based on Y” prompting fail?	Kiedy prompt „dodaj X na podstawie Y” może zawieść?	It fails when Y is outdated, low quality, or architecturally wrong. You may copy hidden flaws unless you validate Y first.	Zawodzi, gdy Y jest nieaktualne, słabe jakościowo albo architektonicznie błędne. Możesz skopiować ukryte problemy, jeśli najpierw nie zweryfikujesz Y.
What minimal Day 4 dashboard metrics would you monitor daily?	Jakie minimalne metryki dashboardu Day 4 monitorować codziennie?	Monitor vectors_saved, collection_count, failures count, and embedding backend availability. These four signals catch most retrieval-readiness regressions early.	Monitoruj vectors_saved, collection_count, liczbę failures i dostępność backendu embeddingów. Te cztery sygnały wcześnie wykrywają większość regresji gotowości retrieval.
